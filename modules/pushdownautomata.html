2.1. Introduction

A Pushdown Automaton (PDA) is the minimal machine that can recognize languages with nested or hierarchical structure using a stack. It is the natural computational counterpart to Context-Free Grammars (CFGs) for parsing balanced patterns and matched constructs.

A PDA can accept a language either by final state or by empty stack, and both acceptance modes are equivalent in power for context-free languages.

Hint

PDAs extend finite automata with a stack memory, giving just enough power to match opens with closes (like parentheses or tags) — but still simpler than a Turing machine.

2.2. Why PDA?

  • DFA/NFA fail on nested patterns because they have only finite memory.

  • PDA succeeds by using a stack — push on every open symbol, pop on each matching close, and accept when both input and stack align.

Note

  • A PDA adds one unbounded LIFO stack to a finite automaton.

  • This lets it match recursive structures such as parentheses, expressions, or HTML tags.

2.3. Everyday Analogy

HTML/XML nesting or balanced parentheses follow the rule: “What was opened last must close first.” That is exactly the LIFO (Last-In-First-Out) discipline of a stack.

Compilers, expression parsers, and markup validators all rely on this PDA principle.

2.4. Formal Definition

A PDA is a 7-tuple:

\[M = (Q, Σ, Γ, δ, q_0, Z_0, F)\]

where:

  • \(Q\) — finite set of states

  • \(Σ\) — input alphabet

  • \(Γ\) — stack alphabet

  • \(δ\) — transition relation

  • \(q_0\) — start state

  • \(Z_0\) — initial stack symbol

  • \(F\) — set of accepting states

A transition \(δ(q, a, X) → (p, γ)\) means: in state q, reading input symbol a (or ε), with X on top of stack — pop X, and push γ (possibly ε) while moving to p.

2.5. Acceptance Modes

  • Acceptance by Final State — consume entire input and end in any state in F; stack contents may remain.

  • Acceptance by Empty Stack — consume input and empty the stack; final state ignored.

Hint

Both modes recognize exactly the same class of languages: Context-Free Languages (CFLs).

2.6. Core Intuition — Balanced Parentheses

  • Push a marker for each “(”.

  • Pop one for each “)”.

  • Reject if a pop is impossible or if the stack is non-empty at end (when using empty-stack acceptance).

This recognizes the classic balanced parentheses language, which DFAs cannot handle.

2.7. Minimal Recipe — PDA for Balanced Parentheses

  • Input alphabet: (, )

  • Stack symbols: (, $ (where $ marks stack bottom)

  • Transitions: - On reading “(”: push (“(”) - On reading “)” with “(” on top: pop - At end: accept by final state or empty stack (via ε-move when only $ remains)

Example Construction

States: q0 (start), q1 (loop), qf (final) Stack start: $

  • q0 –ε, push($)–> q1

  • q1 –‘(’– push(‘(‘)–> q1

  • q1 –‘)’ with ‘(’ on top– pop(‘(‘)–> q1

  • q1 –ε with only $ left– goto qf

Accept either: * by empty stack, or * by final state qf.

2.8. CFG ↔ PDA Equivalence

Every Context-Free Grammar (CFG) has an equivalent PDA, and every PDA corresponds to some CFG. PDAs and CFGs therefore define the same language class: Context-Free Languages (CFLs).

Conversion steps (CFG → PDA): 1. Push the grammar’s start symbol onto the stack. 2. Nondeterministically expand nonterminals by pushing productions. 3. Match input symbols by popping terminals. 4. Accept when stack is empty and input is consumed.

2.10. Why “Pushdown”?

Because its working memory is a pushdown stack — you push when starting a nested construct and pop when finishing it. This mirrors recursive structures and parser stacks in compilers.

2.11. Real-World Uses

  • Parsing programming languages (C, Java, Python)

  • Checking balanced delimiters in code or XML

  • Validating arithmetic expressions using grammar-based stack parsing

Try it Yourself

Design a PDA for \(L = { a^n b^n | n ≥ 1 }\)

Hint: Push a marker for each a; nondeterministically switch to popping phase on first b; Pop one per b, and accept when input ends and stack is empty.

Quick Check

  1. Can a DFA verify “(())”? → ✗ No — requires unbounded matching depth.

  2. What can PDAs do that DFAs can’t? → ✓ Recognize recursive or nested structures.

  3. Which automaton recognizes all CFLs? → ✓ PDA.

  4. Are final-state and empty-stack acceptance different in power? → ✗ No — equivalent.

2.12. Definition 3.1 — Pushdown Automaton

A Pushdown Automaton (PDA) is a tuple:

\[M = (Q, Σ, Γ, δ, q_0, Z_0, F)\]

where δ describes moves that read an input symbol or ε, pop one stack symbol, and push zero or more stack symbols. Acceptance is by final state or empty stack, and both recognize context-free languages.

2.13. Summary

  • PDAs = CFGs in power → both define CFLs

  • DFAs < PDAs < Turing Machines

  • Stack enables recursion, nesting, and matching

  • Foundation for parser design and syntax analysis
